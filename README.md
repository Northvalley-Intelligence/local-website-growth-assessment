# Local Website Growth Assessment

The Local Website Growth Assessment is a North Valley Intel product that helps
local business owners understand why their website may not be generating more
local customers.

It is not a generic SEO grader, security scanner, or technical website scorecard.
The product asks a business-focused question:

> Why is my website not generating more local customers?

The application safely analyzes publicly accessible website evidence, runs an
asynchronous assessment, and produces a plain-English report with explainable
scores, evidence, confidence, limitations, and recommended fixes.

## Current Status

Phase 1, the complete vertical slice, is deployed for founder testing.

- Production: `https://assessment.northvalleyintel.com`
- Staging: `https://staging-assessment.northvalleyintel.com`
- Production source branch: `main`
- Staging source branch: `staging`

The current slice supports URL submission, bounded public crawling, signal
extraction, scoring, evidence-quality checks, report rendering, and customer
demand fit against generated demand data.

## How This Project Uses MDE

This repository is also an applied example of Mission-Driven Engineering, or
MDE. The project is organized around mission progress and validation evidence
rather than just code changes.

### Mission First

The product mission lives in `MISSION.md`. It defines what the system is trying
to achieve, what it is not, what matters most, and which risks are currently
accepted or deferred.

The important product constraint is that reports must be evidence-backed and
business-readable. If the crawl cannot collect enough public evidence, the app
must not invent a confident customer-facing report.

### Artifact Lanes

MDE separates artifacts by audience:

- Human artifacts explain current status and next steps in founder-readable
  language.
- Agent artifacts preserve structured state for future implementation sessions.
- Deploy artifacts record branch, release, and environment facts.

In this repo, those lanes are represented by:

- Human: `MISSION.md`, `MISSION_UPDATES.md`, `docs/operations/mde-status.md`
- Agent: `.mde/state.json`, `.mde/failing-bdds.json`, `.mde/generations/`,
  `.mde/outbox/`
- Deploy: `deploy/release-checks.json`, `deploy/build-manifest.json`,
  `deploy/environment.schema.json`

### Generations And Validation

Implementation work is tracked as generations. Each generation records the
mission focus, behavior changed, validation performed, outcome, remaining risks,
and readiness.

The project uses BDD-style scenarios as part of functional validation, but MDE
does not treat BDDs as the whole quality bar. Validation also includes technical
checks, self-QA against real reports, evidence-quality review, and deployment
checks when a change is released.

Examples of protected behavior:

- Assessments are asynchronous and expose truthful statuses.
- Reports require enough readable public evidence.
- Scores explain found evidence, missing evidence, confidence, and recommended
  action.
- PageSpeed unavailability is reported honestly instead of becoming an invented
  performance score.
- Customer Demand Fit is skipped when the site industry cannot be identified
  confidently.

### Demand Data As A Contract

The assessment app consumes demand intelligence generated by the separate
`demand-data-generation` project. This keeps demand collection separate from
report generation while still allowing reports to compare website evidence
against customer search demand.

The active copied artifact is `packages/shared/src/demand/records.ts`. The app
currently supports demand fit for sectors such as cleaning, real estate,
welding, and senior living when public website evidence is strong enough to
identify the industry.

Monthly search counts are shown only when the upstream demand data provides
them. Null or missing volume is not converted into a fake number.

### Deployment Discipline

Deployment is branch-bound:

- `staging` deploys to `staging-assessment.northvalleyintel.com`
- `main` deploys to `assessment.northvalleyintel.com`

Production should only be deployed from pushed `main` commits after automated
checks, staging validation, and production self-QA have passed.

## Monorepo Structure

- `apps/web`: Next.js app for URL submission, scan status, reports, and API
  routes.
- `apps/worker`: crawling, extraction, PageSpeed adapter, scoring orchestration,
  and report generation.
- `packages/shared`: shared validation, demand fit, scoring rules, and report
  types.
- `.mde`: machine-readable mission state, generation evidence, and outbox sync
  records.
- `docs`: product, engineering, validation, and operations documentation.
- `deploy`: release and environment metadata.

## Local Development

Install dependencies:

```bash
npm install
```

Run the development server:

```bash
npm run dev -w @northvalleyintel/assessment-web
```

Run validation checks:

```bash
npm run format
npm run lint
npm run typecheck
npm run test
npm run test:integration
npm run build
```

Run the Cloudflare/OpenNext build:

```bash
npm run cf:build
```

## Environment

Secrets must stay in local environment files or deployment-provider secret
storage. Do not commit API keys, Cloudflare credentials, PageSpeed keys, admin
passwords, or session secrets.

Relevant environment and release metadata is documented in:

- `deploy/environment.schema.json`
- `docs/engineering/deployment.md`
- `docs/engineering/security-data-handling.md`

## Public Safety

This repository contains product code and MDE process artifacts. It should not
contain customer private data, private credentials, or unpublished client
secrets. Assessment reports are generated from public website evidence and must
state limitations when evidence is incomplete.
